### Fixed (Phase C ŌĆö PR review)
- `MethodPatchView` now ships the source body Pull Original retrieved into `spec.OriginalBody` on Apply, so the snapshot Phase E's diff view will rely on actually persists. The previous flow only filled the body editor; on Apply the spec was created with `OriginalBody = ""`, so after edit + Apply + reload the registry had no record of what the original method looked like. The view now caches the most recent successful Pull keyed by the form's identity (`{Type}::{Method}::{Params}`), invalidates that cache when the form changes (Type / Method / Params edited, Browse-picked a different method, Load loaded a stored spec without an OriginalBody), and copies the snapshot into the spec only when the keys still match ŌĆö never persisting a stale body against the wrong target. Reported in PR review for #14 phase-c.
- `PatchSourcePuller.FindMethodNode` now scopes its candidate set to the declaring type's syntax declarations rather than every `MethodDeclarationSyntax` in the file. The previous walk was `root.DescendantNodes().OfType<MethodDeclarationSyntax>()` filtered only by name + arity, so a file declaring two top-level classes (or an outer class plus a nested helper) that happened to share a method name + parameter count could return the wrong body ŌĆö and the disambiguation step's `return candidates[0]` fallback meant the wrong one would silently win when type names didn't disambiguate (e.g., both are zero-arg). The new path resolves every `TypeDeclarationSyntax` whose simple name + ancestor chain matches the target type (covers nested types and partial classes within one file), then enumerates only those declarations' direct method members. Reported in PR review for #14 phase-c. Verified via probe: in a file containing `class Outer { class Decoy { void Touch(); } void Touch(); }` (decoy declared first in source order), pulling `Outer.Touch` now returns the outer body where the previous code returned the nested decoy's.

### Phase C limitations (intentional, see issue #14 plan)
- Void instance methods only ŌĆö same Phase A scope. Static, non-void, ref/out, generic, constructor, property accessor are all rejected before Pull even runs.
- Block bodies only (`{ ŌĆ” }`). Expression-bodied methods (`=> expr`) are detected and surfaced as a specific error rather than silently returning an empty body.
- Source must live in `Assets/` or `Packages/`. Methods declared in precompiled DLLs (an SDK package without source, Unity-shipped types, etc.) have no MonoScript path to resolve and surface "No MonoScript found".
- Partial classes work (multiple paths walked in order until a matching method node resolves), but auto-generated partials (Unity codegen, source generators) may carry method bodies the user can't easily edit anyway.
- Phase D will add a Roslyn syntax rewriter so the user can write `hp -= amount;` instead of `__set("hp", __get<int>("hp") - amount);` against a Pull'd body. Until then, private member access still requires the helper functions.

### Added (Phase B ŌĆö runtime patch persistence + domain-reload reapply)
- `RoslynRepl.Editor.Patches.PatchPersistence`: project-scoped EditorPrefs store for the runtime patch list. Each spec is six base64-encoded fields (target / method / params / original body / patch body / status) joined by `|`, specs joined by `\n`. Single malformed entry skipped, rest of the set still loads. `LastError` deliberately not persisted ŌĆö it's transient diagnostic state from the last apply attempt.
- `PatchRegistry.AddOrUpdate` / `Remove` / `Clear` now write through to `PatchPersistence` automatically; the in-memory dictionary stays the read path. `PatchRegistry.LoadFromPersistence()` is the inbound counterpart used by the boot path. Authoring-side code (UI, engine) is unchanged.
- `RoslynRepl.Editor.Patches.PatchAutoReapply` (`[InitializeOnLoad]`): on every domain reload, deferred one Editor frame, loads the persisted spec list and re-installs each `Active` patch through `PatchEngine.Apply`. Successful re-applies stay Active; failures flip to `Failed` with `LastError = "Auto-reapply failed: ŌĆ”"` and persist the new status so the next boot doesn't repeat the noise. Inactive / Failed specs are loaded so the UI remembers the user's drafts but are not retried.
- `MethodPatchView`: failed rows now show their `LastError`'s first line as a persistent italic secondary label under the row (`Ōå│ Auto-reapply failed: ŌĆ”`). Hovering still surfaces the full message; the inline preview is one line only so multi-line compile diagnostics don't shift the row layout.

### Phase A limitations (intentional, see issue #14 for the larger plan)
- **Void instance methods only.** Static methods, methods with non-void return types, `ref` / `out` / `in` parameters, generic methods, and constructors are not patchable in MVP. Apply throws a specific diagnostic for each so users know what's gated.
- **No persistence.** Patches survive until the next domain reload (Editor restart, script recompile, or Play Mode toggle on default settings). Phase B will add EditorPrefs persistence + auto-reapply.
- **Helper-style private member access.** Patch bodies reach private fields/methods through `__get<T>("name")`, `__set("name", value)`, `__call<T>("name", args)` rather than the original-source-style `hp -= amount`. Phase D will explore Roslyn-driven auto-rewrite of private references; for the MVP, the helpers stay explicit.
- **No source export.** Phase E adds the diff between the patch body and the original `.cs` source plus an "Apply To File" path; until then patches are purely runtime.

### Fixed (Phase 10 ŌĆö side-effect mitigations)
- `WatchEvaluator.BuildTree` now passes `IncludeProperties = false` to `SimpleObjectSerializer.ToTree`. The result of every Watch refresh used to walk all readable properties of the returned object, calling user-defined getters once per row per Run. Getters in the wild routinely lazy-init, log, mutate counters, or perform IO ŌĆö pinning a single bad watch could amplify those side effects to *N rows ├Ś every Run ├Ś every Editor session*. Watch trees still expand into fields, dictionary entries, and IEnumerable items; only property walk on the result is suppressed. The Output panel's `return X;` rendering still walks properties because that's a one-shot user-initiated inspection, not a per-Run loop. Verified via a counter-getter test object: Output path triggers the getter, Watch path triggers it 0 times.
- `WatchResult` gains a `SourceDescription` field, populated whenever the global-search fallback resolves an expression that didn't compile cleanly. Compile-success rows leave it null because their result is unambiguous. The fallback's `_` carry-over branch tags rows with `previous result (\`_\`)`; the global instance sweep tags rows with the matching entry's display + type + sub-label, e.g. `TitleScene (Anipang3Login) ŌĆö Title` ŌĆö so common-name fields like `count`, `items`, `_data` no longer leave the user guessing which manager they came from. The expression cell renders italic + dimmer for fallback rows (new `rr-watch-cell-expr--global` modifier class), and both expression and value tooltips include the source string. `TryEvaluateGlobalPath` was extended with an `out InstanceEntry matched` parameter so the resolver can hand the picked entry back without relying on indirection through static state.
- `AssemblyReferenceCache` warms up its metadata reference list on a background thread one Editor frame after startup. Previously the first user Run paid for the full `AppDomain.GetAssemblies` walk + `MetadataReference.CreateFromFile` per assembly ŌĆö a 400ŌĆō500ms hang on a typical Unity 6 project (~390 references). Warm-up is best-effort: failures are swallowed, the cache invalidates on `AssemblyLoad`, and the existing lazy-build path inside `GetReferences` remains as a fallback. The defer-by-one-frame avoids extending Editor startup; the background thread avoids blocking the main thread regardless. Verified: cold `GetReferences` measures 476ms / 390 refs, warm read 0ms ŌĆö that's the wall time the user no longer experiences on their first Run after opening the Editor.

### Added (Phase 6 ŌĆö watch panel)

### Added (Phase 6 ŌĆö soft timeout / cancellation)
- `ReplOptions.TimeoutMs` (default 5000ms) wires `CancellationTokenSource.CancelAfter` for every `Execute` call. `ReplOptions.ExternalCancellation` lets a caller (e.g. a future Cancel button or a Watch panel doing batch re-evaluation) thread its own token through; the engine links both via `CreateLinkedTokenSource`. `0` or negative `TimeoutMs` disables the timer.
- `ReplEngine.CurrentCancellation` exposes the linked token to the in-flight snippet; the wrapper class adds a `public static System.Threading.CancellationToken ct` property that pulls through, so snippets cooperate with `ct.ThrowIfCancellationRequested()` inside long loops. `ct` is reset to `CancellationToken.None` between runs so a stale reference can't fire later.
- `ReplResultKind.Cancelled` (and `ReplResult.Cancelled(reason, ŌĆ”)`) distinguishes a clean opt-in cancel from a `RuntimeError`. The reason string disambiguates external cancel vs timeout: "Snippet cancelled by external request" vs "Snippet cancelled after 5000 ms timeout". `RoslynReplWindow` renders cancellations as a `ŌÅ╣` warning line (not red) and the toolbar summary shows "Cancelled".
- The catch-when filter for `OperationCanceledException` accepts both the strict-correct case (token matches) and the common pattern where a snippet manually throws `OperationCanceledException` after observing `ct.IsCancellationRequested` ŌĆö the surrounding token signal is enough to call it a cancel. Other `OperationCanceledException`s thrown from unrelated tokens still surface as `RuntimeError`.

### Fixed (Phase 6 ŌĆö PR review feedback)
- Watch evaluations no longer mutate the user-visible `_` carry-over. Previously every watch ran through `ReplEngine.Execute`, which unconditionally updated `LastResult`, so a watch like `"watch-value"` evaluated between the user's `return 41;` and `return _;` would leave `_` pointing at `"watch-value"` instead of `41`. Added an opt-out `ReplOptions.UpdateLastResult` flag (default `true` for backwards compat) and `WatchEvaluator` sets it to `false`, so passive observation can never quietly mutate the user's previous-result state. Verified with the exact PR repro: `return 41;` ŌåÆ watch eval `"watch-value"` ŌåÆ `return _;` now returns `41`.
- `WatchPanelView` is now disposed before being overwritten on `CreateGUI` rebuild, and disposed again in `OnDisable`. Previously `BindControls` constructed a fresh view on every rebuild without dropping the old one, and the old view's static `WatchStore.Changed` subscription leaked forever ŌĆö every Add/Remove fired N detached panels' refresh handlers and made each watch evaluate N times against the same `WatchStore`. Verified the `WatchStore.Changed` invocation list now stays at 1 across three back-to-back `CreateGUI` calls (was 1ŌåÆ2ŌåÆ3ŌåÆ4), drops to 0 on `OnDisable`, and rises back to 1 on re-mount.

### Phase 6 limitations (intentionally not shipped)
- **`async` / `await` inside snippets.** A first attempt that switched the wrapper signature to `async Task<object>` and `GetAwaiter().GetResult()`'d the task on the Editor main thread reliably deadlocked: most Unity-side awaits post their continuation back to the same main thread that the engine was blocking, so the continuation could never run. Working around this would require rewriting `Execute` as a polling pump on `EditorApplication.update` (with the UI rebuilt around per-frame callbacks instead of a synchronous result), which is out of scope. Snippets that need to observe an async result should `Task.Run` the work themselves and surface a sentinel via `Debug.Log` or the `_` carry-over.
- **Hard-killing a synchronous snippet that ignores `ct`.** `Thread.Abort` is unavailable on Mono / .NET 6+, and the snippet runs on the main thread, so the engine can't unilaterally stop a tight `while(true){}` loop that never checks `ct`. Soft cancel via `ct` is the only safety net; hangs from non-cooperative code still require killing the Editor process.

### Added (Phase 5 ŌĆö persistence + variable continuity)
- `RoslynRepl.Editor.Core.ReplEngine.LastResult` (and `ResetLastResult()`): the most recent successful, *non-null* return value from `Execute`. Synthetic `return null;` fallbacks (snippets without an explicit return) and explicit `return null;` no longer overwrite the carry-over, so a debug-print run won't erase the value the user is iterating on.
- `ReplCodeWrapper` exposes a static property `_` on the generated `__ReplScript` class that pulls through to `ReplEngine.LastResult`. Snippets reference it unqualified (`return _ + 1;`) the same way Python's REPL or ipython surface the previous result. A property ŌĆö rather than an injected local ŌĆö avoids the CS0219 "declared but never used" diagnostic when the user doesn't reference it, and lets a user-introduced local `_` shadow it cleanly inside its own scope (`int _ = 99; return _;` still compiles).
- `RoslynRepl.Editor.Core.ProjectScopedPrefs.BuildKey(baseName)`: extracted helper that namespaces an `EditorPrefs` key with a stable FNV-1a hash of `Application.dataPath`. `UsingsStore` was refactored to use it; `RunHistoryStore` and the upcoming snippet library use the same helper, so adding a new persistence bucket is a one-liner instead of duplicating the hash logic.
- `RoslynRepl.Editor.Core.RunHistoryStore`: project-scoped, `EditorPrefs`-backed ring of recently-executed snippets (capacity 50, most-recent-first). `Push` de-dupes against the head so re-running an identical snippet doesn't churn the ring; older identical entries are kept since they record real moments the user ran something different in between. Entries are base64-encoded before being joined with `\n` so newlines, commas, and quotes inside user code never corrupt the storage format.
- `RoslynRepl.Editor.UI.RunHistoryWindow`: utility popup with a virtualizing `ListView` of history rows (first line as preview + italic "N lines ŌĆó M chars" meta). Double-click loads the snippet into the host editor through a static `OnSnippetChosen` event; a `Clear` button (with confirmation dialog) wipes the project's history.
- Toolbar: new `HistoryŌĆ”` button next to `UsingsŌĆ”`. `RoslynReplWindow.Run` now calls `RunHistoryStore.Push(code)` after `ReplEngine.Execute` returns ŌĆö including failures, since users often want to scroll back to a broken run, fix the error, and retry.
- `RoslynRepl.Editor.Core.SnippetEntry` + `SnippetStore`: project-scoped library of *named* snippets. `Save(name, code)` upserts in place (preserving order on overwrite); `Delete`, `Rename`, and `Exists` round out the CRUD. ### Fixed (Phase 5 ŌĆö PR review feedback)
- `_` is now typed `dynamic` instead of `object` on the wrapper class, so operators and member access bind at runtime against the live value: `return _ + 1;` works directly after `return 41;`, no user-side cast required. Previously the static `object` typing forced CS0019 because `+` doesn't exist on `object` ├Ś `int`. `AssemblyReferenceCache.BuildReferences` force-loads `Microsoft.CSharp` (the C# runtime binder) before snapshotting the AppDomain ŌĆö without that step the binder lives in a not-yet-loaded assembly and the dynamic operator dispatch fails to compile.

Previously the popup snapshotted the store at mount and silently went stale: running snippets while the popup was open updated the backing store but not the visible list. Verified: with the popup already open, three sequential `Push` calls move both `store.Count` and `list.itemsSource.Count` from 0ŌåÆ1ŌåÆ3 in lockstep, with the head row showing the most recent entry.

### Fixed (Phase 4 ŌĆö PR review feedback)
- `UsingsEditorWindow.CreateGUI` now clears `rootVisualElement` before mounting its layout. `CreateGUI` is not a one-shot hook ŌĆö panel rebuilds and domain-reload recovery can fire it again on the same window ŌĆö and the previous code appended a fresh control tree on top of the old one each time, leaving duplicated sections and stale `_customList` references in the older copies. Verified: three back-to-back `CreateGUI()` invocations now leave `rootVisualElement.childCount` at a steady 6, where it previously climbed by 6 every call.
- `UsingsStore` now namespaces its `EditorPrefs` key with a stable FNV-1a hash of `Application.dataPath`, so each Unity project on the machine gets its own bucket. Previously every project on the same user account shared a single `RoslynRepl.CustomUsings` key, which meant adding a project-specific namespace like `MyGame.Runtime` in one repo would silently inject the same `using` into every other Unity project that imported the package, firing CS0234/CS0246 until the user discovered and removed it. The hash uses FNV-1a rather than `string.GetHashCode` because modern .NET runtimes randomize string hashes per process ŌĆö `GetHashCode` would change the storage location on every editor restart and silently lose the user's saved usings. A one-shot migration moves any leftover value from the old global key into the current project's bucket and then deletes the global key so a different project opening the package next doesn't inherit it; if both keys exist (ambiguous which project the legacy value belonged to), the per-project value wins and the legacy key is discarded.

### Fixed (Phase 3 ŌĆö Play Mode safety, surfaced while browsing real managers)
- `SingletonScanner` now detects **any** self-returning public static member regardless of name (`Instance`, `it`, `I`, `Self`, `Current`, `Singleton`, etc.), so plain C# managers like `LobbyFriendManager.it` are surfaced. Member-type matching accepts both directions: the canonical exact / derived case (`public static Foo Instance` declared on `Foo`) is taken regardless of name, and the base / interface case (`public static IService Current => _instance;` on a class that implements `IService`, or `public static BaseThing Instance = new DerivedThing();` declared on the derived class) is taken when the name is in the standard singleton accessor set ŌĆö guarding against false positives like `public static IDisposable s_dispose;` on unrelated classes. Compiler-generated closure / lambda-cache types (names starting with `<`) are filtered out so they don't flood the list.
- For plain C# *property* singletons whose getter we still won't invoke (lazy-init side effects), a sibling private static **backing field** of the same type is read directly when present ŌĆö covering the canonical `private static T _instance; public static T it => _instance ??= new T();` pattern.
- `InstanceEntry.Object` (UnityEngine.Object) ŌåÆ `InstanceEntry.Value` (object). Plain C# instances now flow through the browser and into `ToTree` on double-click.
- `InstanceCategory.All` no longer includes the Singleton sweep ŌĆö that path scans every loaded user assembly and is too heavy to run automatically. Users opt in by selecting the Singleton category.
- `ObjectBrowserView` no longer auto-`Refresh()`s on construction; the first scan happens only when the user changes category, types in search, or presses Ōå╗. Avoids freezing the Editor on window open or domain reload.
- `SimpleObjectSerializer` hardened against runaway graphs:

### Fixed (Phase 2 ŌĆö surfaced while inspecting real Unity scene objects)
- `ValueFormatter.Format` no longer throws `NullReferenceException` when the value is a Unity "fake-null" reference (a wrapper whose native side is destroyed or was never assigned). Each `UnityEngine.Object` / `Component` / `GameObject` branch now compares against null using Unity's overload first and returns a `(destroyed)` marker.
- `SimpleObjectSerializer.BuildNode` short-circuits on fake-null `UnityEngine.Object` values, emitting a `(missing/destroyed)` leaf instead of attempting to walk a destroyed object's fields.
- `UnityEngine.Transform` (and `RectTransform`) is now treated as a leaf type. Its computed accessors (`position`, `lossyScale`, `eulerAngles`, ŌĆ”) read from internal matrices and fire a native `Assertion failed: 'ValidTRS()'` when the matrix is degenerate; those asserts bypass managed `try/catch` and spam the Console.
- Properties whose **declaring type** lives in a Unity-shipped assembly (`UnityEngine.*`, `UnityEditor.*`, `Unity.*`) are skipped. Many of those accessors (`Image.mainTexture`, `Material.color`, `Renderer.bounds`, `Canvas.worldCamera`, ŌĆ”) read native state and trigger the same class of native asserts. Properties declared on **user types** (MonoBehaviour / ScriptableObject subclasses, etc.) are walked normally, so user-defined `Computed => ŌĆ”` accessors stay visible. Users wanting a specific Unity-shipped computed value can call it directly (e.g. `return rect.localPosition;`).
- `BuildDictChildren` now mirrors `BuildEnumerableChildren`'s try/catch: a custom `IDictionary` whose `GetEnumerator()` / `MoveNext` / `Current` throws no longer aborts the whole `ToTree`; partially-collected entries plus a `<enumeration>` error leaf are returned.
- `ValueFormatter.Format` is now guaranteed not to throw. The dict / collection preview branches read `Count` inside their own `try/catch` and fall back to `(IDictionary, count unavailable)` / `(ICollection, count unavailable)`; an outer `try/catch` around the whole formatter returns `<TypeName> <preview error: ŌĆ”>` for any other escape. This closes a hole where `BuildNode` called `Format(value)` *before* `BuildDictChildren`'s guard could run, so a dict with a throwing `Count` getter still aborted the whole `ToTree`.
